1.1.1 Полное наименование системы

1.2 Назначение и цели создания системы

1.2.1 Назначение системы
Система предназначена для обеспечения обмена файлами между пользователями без использования централизованных серверов хранения данных, на основе peer-to-peer архитектуры с возможностью шифрования передаваемых файлов.

1.2.2 Цели создания системы
Система создаётся с целью разработки и исследования архитектуры децентрализованного обмена файлами, обеспечивающей:
• отсутствие централизованного узла хранения файлов;
• идентификацию пользователей на основе криптографических ключей;
• шифрование передаваемых данных;
• устойчивость к сетевым сбоям и временной недоступности узлов;
• возможность расширения до распределённой mesh-архитектуры.

1.3 Описание объекта автоматизации
Объектом автоматизации является процесс установления соединения между пользователями и обмена файлами с обеспечением:
• криптографической аутентификации сторон;
• согласования сеансового ключа для шифрования данных;
• шифрования и проверки целостности файлов;
• подтверждения успешной передачи файлов;
• повторной передачи файлов при временной недоступности узла.

1.4 Требования к системе

1.4.1 Требования к системе в целом
Система должна обеспечивать:
• функционирование в peer-to-peer режиме без внешнего сервера хранения;
• установление защищённого канала передачи данных между двумя узлами;
• передачу файлов в зашифрованном виде;
• подтверждение успешной передачи файлов;
• устойчивость к сетевым сбоям и недоступности узлов.

1.4.1.1 Требования по эргономике и технической эстетике
Интерфейс системы должен быть минималистичным, понятным и ориентированным на демонстрацию функциональности протокола обмена файлами.

1.4.2 Требования к функциям (задачам), выполняемым системой:

Общие функции:
• Установление сетевого соединения между двумя узлами
• Генерация и хранение криптографической идентичности пользователя
• Обмен публичными ключами
• Выполнение криптографического рукопожатия
• Согласование сеансового ключа
• Передача зашифрованных файлов
• Подтверждение успешной передачи файлов
• Повторная отправка файлов при отсутствии подтверждения
• Проверка целостности переданных файлов

Перспективные функции:
• Обнаружение узлов в локальной сети
• Поддержка передачи больших файлов с разбиением на блоки
• Персистентное хранение очереди недоставленных файлов
• Расширение до mesh-сети

1.4.3 Требования к видам обеспечения

1.4.3.1 Требования к лингвистическому обеспечению системы
Средством взаимодействия пользователя с системой является русский язык.

1.4.3.2 Требования к входным и выходным данным

Входные данные:
• Имя пользователя (идентификатор запуска узла)
• IP-адрес и порт удаленного узла
• Выбор файла для передачи

Выходные данные:
• Статус установления соединения
• Статус передачи файлов
• Диагностические сообщения о состоянии соединения
• Подтверждение успешной передачи файлов

1.4.3.3 Требования к программному обеспечению системы
Программное обеспечение должно:
• Реализовывать peer-to-peer архитектуру без внешнего сервера хранения файлов
• Использовать современные криптографические примитивы (Ed25519, Curve25519, симметричное шифрование с аутентификацией)
• Обеспечивать шифрование передаваемых файлов
• Обеспечивать проверку целостности и подлинности данных
• Реализовывать механизм подтверждения передачи и повторной отправки файлов

1.6 Порядок контроля и приёмки системы
Система разрабатывается с применением тестирования работоспособности функционала.
Проверяется:
• Установление соединения между узлами
• Корректность выполнения криптографического рукопожатия
• Корректность шифрования и дешифрования файлов
• Корректность механизма подтверждения передачи
• Устойчивость к временной недоступности узлов
